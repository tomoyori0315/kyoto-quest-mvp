## 0. 30秒サマリ

- **何を作ってるか**：京都・伏見稲荷大社を舞台にした、スマホ縦画面の探索型Web RPG「京都クエスト」。観光情報サイトではなく、歩く・調べる・話す・猫をなでる・帳面を埋める、という探索体験で京都の文化と歴史に親しんでもらうのが狙い。
- **今あるもの**：単一HTMLファイルの動くMVP（`kyoto-quest.html`）。3シーン（大鳥居前→本殿前→千本鳥居）を移動でき、NPC会話・猫なで・調べる・帳面記録・豆知識・エリア遷移がひと通り動く。背景・キャラ・猫はすべて Canvas にコードで描いている（外部ライブラリなし）。
- **次にやること**：背景を「1枚絵のPNG素材」に差し替えられる構造へリファクタし、ビジュアルの完成度を上げる。キャラ・猫・UIは今まで通りコードで別レイヤーとして上に重ねて動かす。
- **最重要の設計判断**：→ 3章を必ず読むこと。背景をPNGにすると「歩ける範囲」「対象の配置」を**別途データ（JSON）で持つ必要がある**。これが固定背景型の肝。

## 1. プロジェクトの目的（仕様のエッセンス）

詳細仕様書は別途あり（v0.2、全20章）。ここでは実装に効く要点だけ抜く。

### 体験の3本柱
1. 世界を歩いている感覚（観光情報を読むのではなく、神社を歩く）
2. 情報・作業・歴史的な学び（調べる/話すと自然に知識が得られる）
3. ちょっとした快楽（猫発見・帳面が埋まる・豆知識・参拝記録などの軽い報酬）

### ビジュアル方針
- スマホ縦画面、9:16
- 見下ろし〜俯瞰寄りの疑似ドット絵
- 温かい和風・箱庭感。色は朱・木・石畳・緑・和紙・黄土色が基調
- UIは木札・和紙・丸ボタン・帳面・スタンプ帳のような温かみ
- 神社や信仰対象を茶化さず、親しみやすく丁寧に扱う
- キャラは2〜3頭身の小さなスプライト

### ゲーム形式：固定背景＋局所自由移動型
- 1エリア＝縦画面の背景1枚。その上に主人公・NPC・猫・UIを別パーツで重ねる
- 十字キーで主人公を一定範囲動かせる。対象に近づくと「話す/調べる/なでる/お参り」ボタンが反応
- 画面端や出口で次エリアへ遷移

### 背景の重要ルール（仕様書5章）
背景画像には **主人公・NPC・猫・UI・会話ウィンドウを描き込まない**。
理由：描き込むと動かせないから。これらは必ず別レイヤーで実装する。
→ この方針が、今回のリファクタ方向そのもの。

### テキスト方針（学び要素）
説明文を押し付けない。会話や調査で自然に出す。関西弁の温度感で。
- 悪い例：「伏見稲荷大社は和銅4年に創建され……」
- 良い例：「ここは“お稲荷さん”として親しまれてる神社やで。商売繁盛や五穀豊穣を願って来る人も多いんや。」
- 詳しい説明は「もっと読む」で展開する（実装済み）。

### 固有名詞の扱い
- 神社・地名など実在の固有名詞はOK（伏見稲荷大社、千本鳥居、本殿、楼門 等）
- 猫は実在の猫を特定せず、ゲーム内の架空の猫として扱う
- ご朱印は信仰と結びつくので雑にアイテム化しない。「参拝記録/旅の記録/記念印」として、リアルなご朱印とは別物に

## 2. 現状の `kyoto-quest.html` の構造

単一HTMLファイル。`<style>` にUI、`<script>` にゲーム全体。外部ライブラリなし。
内部解像度 180×320 の Canvas を CSS で拡大（`image-rendering:pixelated`）。

### 主要な変数・データ
| 名前 | 役割 |
|---|---|
| `VW, VH` | Canvas内部解像度（180×320） |
| `SCENE` | 現在のシーン `'sando' / 'romon' / 'senbon'` |
| `WORLD_H` | そのシーンのワールド縦サイズ（カメラ範囲） |
| `floorTop() / floorBot()` | シーンごとの「歩ける縦範囲」。**ここが当たり判定の上下限** |
| `P` | プレイヤー状態 `{x,y,dir,moving,anim,speed}` |
| `game` | 進行状態 `{started,cats,spots,visits,quest,nearest}` |
| `ENTITIES` | 現在シーンの対象配列。`type: npc/cat/object/exit` |
| `SANDO_ENTITIES` | 参道シーンの対象定義（他シーンは `enterRomon/enterSenbon` 内に直書き） |

### エンティティの形（重要）
```js
// NPC
{id, type:'npc', x, y, r, sprite:{...}, name, act:'はなす',
 lines:[{t:'本文', learn:'<b>豆知識</b>...', spot:'記録ID'}, ...]}
// 猫
{id, type:'cat', x, y, r, sprite:{body,calico}, name, act:'なでる',
 react:'なでた時の一言', learn:'猫帳に残る説明'}
// 調べる対象
{id, type:'object', x, y, r, name, act:'しらべる',
 lines:[{t, learn, spot}], visit:'参拝ID'(本殿などお参り対象のみ)}
// 出口
{id, type:'exit', x, y, r, name, act:'すすむ/もどる', to:'遷移先SCENE'}
```
- `x,y` = 判定の中心座標、`r` = 反応する半径
- `spot` = 見聞（調べた記録）のID、`visit` = 参拝記録のID
- 近接判定 `updateNearest()` は「半径内で一番近い1つ」を拾う。
  → **対象同士を近づけすぎると片方が隠れる**（実際にこれでバグった。3章参照）

### 主な関数
- 背景描画：`drawSando / drawRomon / drawSenbon`（＋部品 `bigTorii, romonGate, hondenSprite, lantern, foxStatue, shopsLeft, rightSide, stonePath, temizu, trees`）
- キャラ：`drawPlayer, drawNPC, drawCat`
- 当たり判定：`canWalk(x,y)`（`floorTop/Bot` と `blockers` 配列でブロック）
- 近接：`updateNearest()`
- 会話：`startDialog / showPage / advance / endDialog`
- 遷移：`goExit / enterSando / enterRomon / enterSenbon`
- 記録・UI：`recordVisit, updateCatCount, setQuest, toast, openBook, renderBook`
- ループ：`updatePlayer → updateNearest → render`（`requestAnimationFrame`）

### 操作
- 画面：十字キー（pointer）＋丸ボタン（近接対象でラベルが変化）＋📒帳面
- PC確認用キーボード：矢印移動 / Z・Enter・Space＝アクション / B＝帳面

## 3. ★今回ハマった点と設計判断（一番大事な引き継ぎ）★

### (1) コードで全部描く方式の限界
背景・建物・キャラを全部 Canvas にコードで描いている。雰囲気は出せるが、
**完成イメージ（描き込まれた一枚絵のドット）の密度には届かない**。
瓦・提灯・暖簾の文字・木々の陰影みたいな描き込みを、コードで手描き相当に起こすのは費用対効果が悪すぎる。
小さいスプライト（特に狐像）は、簡略化するとウサギっぽく見えてしまう等の限界もある。

→ **結論：背景は画像素材（PNG）にするのが正解。** キャラ・猫・UIはコードのまま別レイヤーで重ねる。
この分業なら、背景のクオリティを上げつつ、動く部分は柔軟に保てる。仕様書5章の方針とも一致。

### (2) 背景をPNGにすると「判定データ」を別で持つ必要がある（最重要）
今はコードで背景を描いてるから、建物や歩ける範囲の座標をプログラムが知っている。
**背景を1枚絵の画像にすると、プログラムは絵の中身を理解できない。**
だから背景画像とは別に、以下を**座標データ（JSON）として持つ**必要がある：
- 歩ける範囲（walkable area / 当たり判定）
- NPC・猫・調べる対象の立ち位置と反応半径
- 出口の位置と遷移先

これが固定背景型ゲームの肝。→ JSON形式案は6章。

### (3) 近接判定は「一番近い1つ」しか拾わない → 対象を離して置く
出口を2つ近くに置いたら、手前の1つが常に優先されて、奥の出口（千本鳥居へ進む）が
反応せず「先に進めない」状態になった。
→ 対策：**前に進む出口は奥（画面上）、戻る出口は手前（画面下）**に、はっきり離して配置。
判定が重なる配置は避ける。これはデータ設計のルールとして 6章/7章 に反映済み。

### (4) 判定の上下限はシーンごとに違う
`floorTop()/floorBot()` がシーンで異なる。対象の `y` がこの範囲の外（特に上）にあると、
近づけず反応しない。本殿が歩ける範囲の上限より上にあって届かない、というバグが起きた。
→ **対象の判定中心は、必ず歩ける範囲から半径内に届く位置に置く。**

## 4. 開発ロードマップ（この順番で進めるのが安全）

### Step 0：リポジトリ整備 + ルールファイル
- gitリポジトリ化。`AGENTS.md`（Codex向けルール、5章の叩き台）と `README.md` を置く
- `kyoto-quest.html` を起点に置く

### Step 1：背景差し替え可能な構造へリファクタ（背景はまだコード描画でOK）
- 「背景を描く処理」と「キャラ・判定・データ」を分離
- マップ1枚分のデータ（歩ける範囲・エンティティ配置・会話）を**JSONに切り出す**（6章の形式）
- シーン定義をコード直書きから、データ駆動（JSON配列をループで生成）に変える
- ゴール：`enterRomon` 内に直書きしてる対象配列が、JSONを読むだけで構築される状態

### Step 2：背景PNGを1枚だけ差し込む（大鳥居前だけ）
- 縦長背景PNG（9:16）を1枚用意（AI生成 or 手描き）。**人・猫・UIは描き込まない**
- `drawSando` を「PNGを読み込んで描画」に差し替え（`Image` を `drawImage`）
- JSONの座標と絵がズレてないか、キャラが正しく床に乗るか確認
- **1枚で検証してから横展開する**（いきなり全シーン作るとズレ時に全やり直し）

### Step 3：（余裕が出たら）配置エディタ
- 背景PNGの上でクリックして「歩ける範囲」「対象位置」を置き、JSONを吐き出す簡易ツール
- シーン追加が一気に楽になる。後回しでよい

### Step 4：シーン拡張
- 同じ仕組みで JR稲荷駅前 → 表参道入口 → 裏参道 → 京阪方面 … を追加
- 仕様書6章のエリア候補に沿って

### 背景AI生成について（沼ポイント注意）
背景の生成は実は一番むずかしい工程。安定して出すには次を満たす必要がある：
- 人・猫・UI・会話ウィンドウを描き込まない
- 歩ける床がちゃんとある構図
- 9:16の縦構図
- 複数シーンで画風（色味・タッチ・パース）を揃える
→ ここは専用にプロンプトを詰める価値あり。1枚成功したらそれを基準に画風を固定する。

## 6. マップ JSON 形式案（Step 1〜2 で使う）

背景画像と判定データを分離するための形式。1シーン=1ファイル（`data/maps/sando.json` など）を想定。
座標は「背景画像のピクセル座標」を基準にすると分かりやすい（内部解像度に合わせてスケールしてもよい）。

```json
{
  "id": "sando",
  "name": "伏見稲荷 大鳥居前",
  "background": "assets/bg/sando.png",
  "size": { "w": 180, "h": 380 },
  "spawn": { "x": 90, "y": 300, "dir": "up" },
  "walkable": {
    "type": "rect",
    "top": 196, "bottom": 352, "left": 34, "right": 146
  },
  "blockers": [
    { "x": 52, "y": 206, "w": 14, "h": 12, "note": "左の狐像" },
    { "x": 128, "y": 206, "w": 14, "h": 12, "note": "右の狐像" }
  ],
  "quest": "楼門にお参りしよう",
  "entities": [
    {
      "id": "guide", "type": "npc", "x": 120, "y": 300, "r": 22,
      "name": "案内のおじさん", "act": "はなす",
      "sprite": "guide",
      "lines": [
        { "t": "ようおまいり！ここが伏見稲荷の大鳥居やで。でかいやろ〜。" },
        { "t": "鳥居をくぐる時はな、端を通って軽く一礼。真ん中は神さまの通り道やで。",
          "learn": "鳥居は俗世と神域の境目。くぐる時に気持ちを切り替えるんやて。",
          "spot": "torii_manner" }
      ]
    },
    {
      "id": "cat_torii", "type": "cat", "x": 96, "y": 286, "r": 18,
      "name": "気まぐれ猫", "act": "なでる", "sprite": "calico",
      "react": "…ゴロゴロ。(撫でられて満足そう)",
      "learn": "鳥居の根元がお気に入りの昼寝スポットらしい。"
    },
    {
      "id": "obj_torii", "type": "object", "x": 90, "y": 236, "r": 24,
      "name": "大鳥居", "act": "しらべる",
      "lines": [
        { "t": "見上げるほど大きな朱色の鳥居。くぐると、いよいよ神さまの領域や。",
          "learn": "朱色は魔除けと生命力の象徴。連なる朱の鳥居が伏見稲荷の名物やね。",
          "spot": "big_torii" }
      ]
    },
    {
      "id": "exit_romon", "type": "exit", "x": 90, "y": 200, "r": 18,
      "name": "楼門へ進む", "act": "すすむ", "to": "romon"
    }
  ]
}
```

補足：
- `walkable` はまず矩形(rect)で十分。複雑な形が必要になったらポリゴンやマスク画像に拡張
- `sprite` は文字列キーにして、コード側のスプライト定義テーブルと対応づける（背景がPNGでもキャラはコード描画のため）
- `spot/visit` のIDは、帳面（見聞/参拝）に記録するキー。表示用メタ（タイトル・本文）は別テーブル `data/records.json` 等にまとめてもよい
