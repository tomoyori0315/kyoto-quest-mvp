# ASSET_SPEC.md - 京都クエスト 背景アセット仕様

## 目的

この文書は、`assets/bg/sando.png` などの背景PNGを作る時の基準をまとめるものです。
背景はゲーム内の「1シーン分のワールド画像」として扱い、主人公・NPC・猫・UIは別レイヤーで描画します。

## 基本仕様

- 画像形式: PNG
- 配置先: `assets/bg/`
- 座標基準: Canvas内部解像度と同じ `VW = 180`
- 表示窓: `180 x 320`
- 背景画像の高さ: 各シーンの `MAPS[sceneId].worldH` に合わせる
- 背景画像には、歩ける範囲・対象配置・出口判定を直接持たせない
- 歩ける範囲、ブロッカー、NPC、猫、調べる対象、出口は `MAPS` 側の座標データで管理する

重要: スマホ画面は9:16ですが、背景PNGそのものは必ずしも `180 x 320` ではありません。
現在の実装では、画面の表示窓が `180 x 320`、その奥に `worldH` 分の縦長背景があり、カメラ `camY` で一部を切り出して見せています。

## 推奨背景サイズ

| sceneId | ファイル | worldH | 推奨サイズ | 備考 |
|---|---|---:|---|---|
| `sando` | `assets/bg/sando.png` | 380 | `180 x 380` | 大鳥居前。9:16の `180 x 320` ではなく、縦に少し長いワールド画像として作る |
| `romon` | `assets/bg/romon.png` | 380 | `180 x 380` | 楼門・本殿前。上側に本殿、下側に歩ける余白を持つ |
| `senbon` | `assets/bg/senbon.png` | 400 | `180 x 400` | 千本鳥居。奥行きのある鳥居列を縦長に描く |

## 高解像度制作時の目安

ゲーム内の座標基準は `180 x worldH` です。
一方で、AI生成や画像編集で背景を作る時は、同じ比率の高解像度版で制作してかまいません。

| sceneId | ゲーム内基準 | 5倍制作サイズの目安 |
|---|---|---|
| `sando` | `180 x 380` | `900 x 1900` |
| `romon` | `180 x 380` | `900 x 1900` |
| `senbon` | `180 x 400` | `900 x 2000` |

画像比率は必ず `180:worldH` に合わせてください。
比率が違う画像をそのまま使うと、Canvas描画時に `VW x map.worldH` へ引き伸ばされ、横または縦に歪む可能性があります。

## 描き込んではいけないもの

背景PNGには、動かすもの・UIとして重ねるものを描き込まないでください。

- 主人公
- NPC
- 猫
- UI
- 会話ウィンドウ
- デバッグ表示

これらを背景に描き込むと、移動・会話・猫帳・デバッグ確認・将来の配置調整ができなくなります。

## 描き込んでよいもの

背景として固定されるものは描き込んでよいです。

- 建物
- 鳥居
- 石畳
- 店舗
- 灯籠
- 木々
- 看板
- 柵
- 階段
- 影や光

ただし、看板などを調べる対象にする場合でも、反応位置と半径は背景画像ではなく `MAPS.entities` 側で管理します。

## 描画実装との関係

現在の背景画像描画は、`kyoto-quest.html` 内で次の形です。

```js
ctx.drawImage(bg.img, 0, -camY, VW, map.worldH);
```

つまり、読み込まれた背景画像は `VW x map.worldH`、現在なら `180 x worldH` に引き伸ばして描画されます。
画像の比率が `VW:worldH` と違う場合、横または縦に歪む可能性があります。

そのため、背景PNGは各シーンの推奨サイズと同じ比率で作るのが安全です。
例えば `sando` は `worldH: 380` なので、`180 x 380` 基準で作ってください。

## 制作時の確認ポイント

- 背景内に主人公・NPC・猫・UIが描き込まれていない
- 歩ける床が画面下から奥へ自然につながっている
- `MAPS.walkable` の範囲に合わせて、プレイヤーが立てる場所が絵として自然に見える
- `MAPS.blockers` の位置に、灯籠・看板・狐像などの障害物がある
- `MAPS.entities` の判定中心が、歩ける範囲から反応半径内に届く
- 前に進む出口は奥、戻る出口は手前に見える構図になっている
- デバッグ表示をONにした時、判定円と絵の対象位置が大きくズレていない
